Critical correction · agent teams

The Agent Teams env var is widely typed wrong

The env var that enables Claude Code's Agent Teams feature is CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1, not CLAUDE_CODE_EXPERIMENTAL_TEAM=1. The shorter form does not exist in the official docs and is widely typed wrong across blog posts and gists. If ~/.claude/settings.json contains the typo, Agent Teams is silently disabled — no error, no warning, the feature simply does not turn on.

Set it in ~/.claude/settings.json:

{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}

Or as a shell env var before launching Claude Code:

export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
claude

The /btw command mid-task side-questions without polluting history commands

Shipped in Claude Code 2.1.72 (March 2026). Lets you ask Claude a mid-task question — "by the way, what does this regex do?" — without writing it into the main conversation history. Reports cite up to ~50% total token savings in sessions where the user asks lots of side questions, because the side-thread doesn't pile up in the next turn's context window.

Usage is the same shape as any slash command:

/btw what's the difference between Promise.all and Promise.allSettled?

Boris Cherny says he uses it constantly — it's the cheap way to keep one main thread on-rails while still getting Claude's help with the questions that pop into your head while reading its output.

Permission modes (v2.1) default · acceptEdits · plan · auto · dontAsk · bypassPermissions commands

Six permission modes in v2.1. auto and dontAsk were added in 2026 — they sit between acceptEdits (auto-approve edits but still gate everything else) and bypassPermissions (the nuclear "never ask me anything" mode).

Enter Plan Mode from anywhere with Shift+Tab Shift+Tab.

modebehaviorcopy
defaultPrompts on every action that isn't pre-allowed. Safest, slowest.
acceptEditsAuto-approves file edits; still prompts for shell commands and other tools.
planRead-only investigation; Claude writes a plan, no edits or commands run. Enter with Shift+Tab Shift+Tab.
auto2026 addition. Runs unattended for longer stretches; combines acceptEdits behavior with quieter prompting policy.
dontAsk2026 addition. Never prompt — for tools matching an allowlist. Tighter than bypassPermissions because it's scoped to allow rules.
bypassPermissionsSame effect as --dangerously-skip-permissions: pre-approve everything. Containers only.

Hook system extensions (2026) HTTP hooks · defer outcome · additionalContext · baseline hooks hooks

The hook system in 2026 is meaningfully more capable than the 2025 version. Three changes matter most: hooks can fire over HTTP, PreToolUse outcomes include a new defer verb, and PreToolUse output can inject context mid-decision via additionalContext.

HTTP hooks (Feb 2026)

since v2.1.x

Hooks can now fire against a webserver instead of a local script. Useful for shared org-wide policies (secrets-scanning, audit logging, license-compliance checks) that don't want to be checked into every repo's .claude/settings.json.

PreToolUse outcomes: allow · deny · ask · defer

defer requires v2.1.89+

The hook's JSON response decides what happens to the in-flight tool call. The three classic outcomes (allow, deny, ask) are joined by defer: postpone the decision until another hook or a user gate weighs in. Lets you build chained policies without saying "yes" or "no" prematurely. 

{
  "outcome": "defer",
  "additionalContext": "Bash command targets a production host — wait for SRE sign-off."
}

additionalContext field in PreToolUse output

v2.1.9+

A free-text field your hook can populate to inject context into Claude's reasoning mid-decision. Use it to surface the "why" behind a deny ("this file is in a protected path") so Claude adapts instead of just retrying the same call.

Baseline recommended hooks

low-effort, high-leverage

Two hooks worth setting up on day one:

  • PreToolUse secrets-scan before Edit/Write — block any change that would write a file matching a secrets pattern (API keys, private keys, .env contents).
  • PostToolUse lint trigger after Edit/Write — run your project's linter on the changed file so Claude sees the failures and self-corrects without waiting for CI.

CLI flags worth knowing resumption · automation · debugging cli-flags

The flag set is small; these four cover most of the leverage.

--dangerously-skip-permissions Pre-approve every permission prompt for the entire session. Equivalent to permission mode bypassPermissions. Containers only — don't aim it at your real home directory.
--mcp-debug Verbose MCP logs. First thing to reach for when an MCP server is connecting but its tools don't show up, or when a tool errors mysteriously.
--continue Resume your most recent session.
--resume <session_id> Resume a specific session by ID.
--print --output-format json One-shot, scripting-friendly output. Run Claude inside a shell pipeline and get parseable JSON back.

settings.json power moves experimental.* · disabling MCPs · permission patterns · env settings.json

Four patterns that pay back the first day you adopt them. Settings live in ~/.claude/settings.json (global) or <project>/.claude/settings.json (per-repo).

experimental.* block

The most likely landing zone for new beta toggles. Worth watching across releases — new flags show up here before they get documented.

{
  "experimental": {
    "featureName": true
  }
}

mcpServers[*].disabled: true

Temporarily silence an MCP server without ripping it out of your config. Re-enable by flipping the bool. Useful when one server is noisy or broken and you want to keep the others running.

{
  "mcpServers": {
    "noisy-server": {
      "command": "noisy-server-binary",
      "disabled": true
    }
  }
}

permissions.allow / permissions.deny

Auto-approve patterns — the structured, surgical alternative to --dangerously-skip-permissions. Allowlist specific commands or tool calls you trust; the rest still prompt.

{
  "permissions": {
    "allow": [
      "Bash(git status)",
      "Bash(git diff *)",
      "Bash(npm test)"
    ],
    "deny": [
      "Bash(rm -rf *)"
    ]
  }
}

env: { ... }

Set env vars per Claude Code session, including the Agent Teams flag from the critical-correction section above. Lives anywhere Claude Code's settings get loaded from.

{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1",
    "CLAUDE_CODE_MCP_INSTR_DELTA": "1",
    "ANTHROPIC_MODEL": "claude-opus-4-7"
  }
}

Boris Cherny's top tips from the engineer who built Claude Code tips

Boris Cherny built Claude Code. The five tips below are the ones he says give the biggest leverage in practice — drawn from his public writeups and the "how Boris uses Claude Code" page.

workflow

3–5 parallel sessions in separate git worktrees

Boris calls this "the single biggest productivity unlock." Use git worktree add to spin up isolated trees and run one Claude per worktree. While one is investigating, another is implementing, a third is reviewing.

workflow

Plan Mode first, then auto-accept

Press Shift+Tab Shift+Tab to enter Plan Mode. Iterate the plan with Claude. Then switch to auto-accept — Claude usually one-shots the implementation cleanly because the thinking already happened.

model

Opus with thinking enabled — for everything

Less correction time, faster total prompt-to-done even when the per-call latency is higher. Set ANTHROPIC_MODEL=claude-opus-4-7 and bump CLAUDE_CODE_MAX_THINKING_TOKENS.

memory

CLAUDE.md as a living document

Every time Claude does something wrong, add a rule. Check it in. Share it with the team. The file becomes shared institutional memory — onboarding for your AI collaborator that improves every week.

commands

Slash commands for every repeated workflow

Drop a markdown file at .claude/commands/<name>.md, check it into git, share it across the team. Anything you've typed twice belongs here — code-review prompts, release checklists, debug rituals.

Unusual / less-known tricks statusline · context-halving · Gemini-as-minion · containers unusual-tricks

The long-tail of community-discovered patterns. Lower confidence than Boris's tips — verify against your own workflow before relying on them.

customization

Custom statusline script

Roll your own status bar showing model, branch, and remaining token budget. Reduces context-switching cost when you're juggling parallel sessions.

context

Halve the system prompt via aggressive CLAUDE.md structuring

Multiple writeups report measurable context savings by tightening CLAUDE.md. The trick is to make every rule earn its keep — short, specific, no preamble.

multi-model

Gemini CLI as a "minion"

Have Claude shell out to gemini-cli via Bash for cheap lookups (file summaries, structured extraction). Keep Opus context reserved for the real reasoning work.

safety

Claude Code in a container

For risky or destructive tasks (mass refactors, scripted migrations, "fix every package.json"), run Claude inside a container with --dangerously-skip-permissions. The blast radius is bounded by the container's filesystem.

context

Tight context reset over long conversations

Instead of running one long thread, start a fresh session with a 2–3 line context recap. Reports cite 50–70% token savings at equal output quality — long threads accumulate dead context the model still has to attend to.
